## 1. Ce que la documentation exige (synthese)

### 1.1 Pipeline obligatoire (toute feature)

D'apres `05-workflows/agentic-feature-pipeline.md` et `feature-delivery.md` :

1. Cas d'usage + acteur + criteres d'acceptation atomiques
2. Etats UI (loading, empty, error, denied, stale, reconnect)
3. Transitions cycle de vie
4. Contrats Protobuf avant endpoint
5. Decoupage par couche + tests par couche
6. Aucun timer vers `ACTIVE_ROUND` ; aucune publication score sans commande admin

### 1.2 Gates transversaux (`acceptance-gates.md`)

| Gate | Exigence doc | Code v0.1 |
|---|---|---|
| Produit | Vocabulaire party/participation/round | OK domaine + routes |
| Lifecycle | Pas de demarrage auto par timer | OK (tests domaine) |
| Contrat | Protobuf versionne | OK proto ; RPC 27% implemente |
| Securite | RBAC + participation serveur | Partiel (REST OK ; live partiel) |
| Realtime | Reconnect teste | Partiel (integration test ; pas E2E) |
| Publication | Scores apres publication seulement | OK domaine ; pas d'API/UI cablee |
| Qualite | typecheck, lint, test, build | OK ; docs et E2E Playwright egalement valides |
| UI | Shells par audience, no scroll page | Partiel (shells OK ; donnees mock) |
| Viewport | Scroll dans panneaux seulement | Partiel (CSS global ; pas QA systematique) |

### 1.3 Parcours produit (`use-case-coverage.md`) — 14 flux

Chaque parcours exige variantes atomiques (empty, stale, double-clic, conflit multi-admin,
remboursement, preferences notification, etc.). La doc elle-meme note que les variantes les moins
detaillees sont : mini-jeux un par un, multi-admin, litiges finance, notifications, support complet,
post-publication.

### 1.4 Sprint UI (`sprint-UI/`)

Exige pour chaque sprint 00–19 : routes, wireframes, etats obligatoires, mapping user story → surface.
Le code web couvre ~90% des **routes** inventoriees mais ~15% du **branchement** backend.

## 7. Conclusion

La documentation (`sprints 00–19`, `use-case-coverage`, `sprint-UI`, `04-layers`) decrit un produit
**complet** : 14 parcours narratifs, 11 domaines protobuf, 5 acteurs UI, zero fuite de score provisoire,
admin command center, mini-jeux, workers, compliance.

La codebase v0.1 a **correctement investi en profondeur** sur domaine, persistence, contrats et API REST
du noyau metier (sprints 01–10 en backend). L'**ecart principal** n'est pas l'absence de vision mais :

1. **UI structurelle sans donnees reelles** (~25 fichiers mock)
2. **ConnectRPC a 27%** alors que la doc cible 100%
3. **Scoring/publication inexistant** cote application (sprint 13)
4. **Mini-jeux absents** (sprints 14–15)
5. **Workers et notifications non operationnels** (sprint 17)
6. **Compliance/incidents non modelises** (sprint 18)
7. **Recette E2E quasi vide** (sprint 19)

Le prochain saut de valeur documentaire = **Phase A** : un parcours joueur + admin **sans mock** jusqu'a
publication de scores, meme sans mini-jeu dedie.
